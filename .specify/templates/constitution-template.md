# Dev-Stack Baseline Practices

> **This section is auto-generated by dev-stack.** It documents the non-negotiable
> practices that every contributor — human or AI — must follow in this repository.

## Atomic Commits

Every commit represents **a single logical unit of work**. A logical unit of work
is the smallest set of changes that:

1. **Implements a single task** — one intent, one outcome.
2. **Includes passing tests** — any new or changed behavior has corresponding tests
   that pass in isolation.
3. **Leaves the codebase buildable** — the project compiles/runs correctly at this
   commit, and no broken intermediate state is recorded in version history.

### What is NOT an atomic commit

- Mixing a bug fix with an unrelated refactor.
- Committing test code separately from the production code it covers.
- Committing code that does not build or whose tests fail.

### Practical guidance for AI agents

- Before committing, verify that all relevant tests pass.
- If a task requires changes across multiple files, include them all in one commit
  as long as the intent is singular.
- When in doubt, prefer smaller commits — they are easier to review and revert.

## Commit Message Format

Every commit message in this repository must follow **Conventional Commits** with
structured body sections. The `commit-msg` hook enforces this format automatically.

### Subject line

```
<type>(<scope>): <description>   (max 72 characters)
```

Allowed types: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`,
`ci`, `chore`, `revert`.

### Body sections

Include these sections in order, each introduced with a `##` heading:

1. **Intent** — why this change exists.
2. **Reasoning** — critical decisions and trade-offs.
3. **Scope** — components and files touched.
4. **Narrative** — 3–5 sentences summarizing the change for future AI agents.
   Commit history serves as persistent memory; write narratives that enable
   future context retrieval.

### Trailers (required for agent commits)

If the commit is authored by an AI agent, the following trailers are **required**
at the end of the message body:

```
Spec-Ref: specs/<id>/spec.md
Task-Ref: specs/<id>/tasks.md
Agent: <agent-name>
Pipeline: lint=pass,typecheck=pass,test=pass,...
Edited: false
```

| Trailer | Description |
|---------|-------------|
| `Spec-Ref` | Path to the specification file (relative to repo root) |
| `Task-Ref` | Path to the tasks file (relative to repo root) |
| `Agent` | Name of the AI agent that authored the commit |
| `Pipeline` | Comma-separated `stage=status` pairs from the automation pipeline |
| `Edited` | `true` if a human edited the agent's output, `false` otherwise |

### Manual (human) commits

Human-authored commits only require a valid conventional commit subject line.
Trailers are optional for manual commits — the `commit-msg` hook only enforces
trailers when an `Agent:` trailer is present.

### What the hook rejects

- A subject that does not match `type(scope): description`.
- An agent commit missing any of the five required trailers.
- A `Spec-Ref` or `Task-Ref` path that does not exist in the repository.

## Test-Driven Development

Follow the **Red-Green-Refactor** cycle for every code change:

### 1. Red — Write a failing test first

- Define the expected behavior before writing any implementation.
- Run the test and confirm it fails for the right reason (not a typo or import error).

### 2. Green — Write the minimal implementation to pass

- Write only enough code to make the failing test pass.
- Avoid over-engineering or adding features not required by the current test.

### 3. Refactor — Improve the code while tests remain green

- Clean up duplication, improve naming, simplify logic.
- Re-run all tests after every refactoring step — they must remain green.

### Practical guidance for AI agents

- Never skip the "Red" step: a test that was never seen failing provides weaker
  evidence of correctness.
- Write tests in the same commit as the implementation they cover.
- If working on a bug fix, write a test that reproduces the bug before fixing it.

# Commit Pipeline

End-to-end workflow: smart staging → atomic commit → pipeline resolution → rebase → PR → review → fix loop.

## When to Use

- Committing any set of changes (dev-stack repo or otherwise)
- Unsure what files to stage / what to gitignore
- Needing a well-formed Conventional Commit with body + trailers
- Opening a PR and handling automated Copilot review
- Enforcing TDD red-green-refactor discipline before pushing

## Expected Output

- A clean atomic commit (or series of commits) with Conventional Commit format
- All hard-gate pipeline stages passing (`lint`, `typecheck`, `test`, `security`, `docs-api`)
- PR opened on current branch against `origin/main`
- All Copilot review suggestions evaluated; valid ones implemented and committed

---

## Procedure

### Phase 0 — Context Assessment

Understand the current state before touching anything.

```bash
git status                      # overview of changed/untracked files
git diff --stat HEAD            # files changed with line counts
git diff HEAD                   # full diff for logical analysis
git log --oneline -5            # recent commits for context
```

Also check:
- Is this a **dev-stack repo**? → look for `dev-stack.toml` at project root
- Is there a **constitution**? → check `.specify/memory/constitution.md` (also try `.specify/constitution.md`, `specs/constitution.md`)
- Is there a **spec/task reference**? → check open specs in `specs/` or `.specify/memory/`
- Note `Spec-Ref` and `Task-Ref` for the commit trailers (use `None` if not applicable)

---

### Phase 1 — Smart Staging

**Decide what belongs in this commit.** See [file-staging-rules](./references/file-staging-rules.md) for the full classification guide.

#### 1a. Classify every changed file

For each file, assign it to one category:

| Category | Action |
|---|---|
| Production source + its tests | Stage — core of the commit |
| Config / infra files changed as part of this task | Stage |
| Dev-stack pipeline artifacts (see §1c) | Stage if generated by this commit's pipeline run |
| Secret / credential files | Add to `.gitignore`, never stage |
| Build outputs (`__pycache__`, `dist/`, `.egg-info/`) | Add to `.gitignore` if missing, never stage |
| IDE/OS files (`.DS_Store`, `.idea/`, `.vscode/`) | Add to `.gitignore` if missing, never stage |
| Unrelated changes (different feature/fix) | Do **not** stage — defer to a separate commit |

#### 1b. Enforce atomic commit

Before staging, verify the change is **one logical unit**:
- One intent (one feature, one bug fix, one refactor, one chore)
- Production code and its tests staged together
- No mixing of unrelated changes

If the working tree contains unrelated changes, ask the user which subset to commit first. Do not stage everything blindly.

#### 1c. Dev-stack repos: pre-stage artifact awareness

If `dev-stack.toml` exists, the commit will trigger the dev-stack pipeline hooks.  
Check [dev-stack-pipeline](./references/dev-stack-pipeline.md) to understand which artifacts each stage will generate or mutate — **do not add them before the pipeline runs**, but be ready to stage them afterward.

#### 1d. Update .gitignore if needed

Before staging, add any missing `.gitignore` entries for files that should never be tracked:

```bash
cat .gitignore    # review current rules
# Common additions:
echo "__pycache__/" >> .gitignore
echo "*.pyc" >> .gitignore
echo "dist/" >> .gitignore
echo "*.egg-info/" >> .gitignore
echo ".env" >> .gitignore
echo ".secrets" >> .gitignore
echo "docs/_build/" >> .gitignore
```

Then stage the changed `.gitignore`.

### Phase 2 — TDD Validation

Before committing, confirm the TDD contract holds:

```bash
# Run the full test suite
pytest -x -q    # (or whatever test command is configured)
```

**Hard rules** (do not proceed if any fail):
1. Tests pass — zero failing tests
2. New/changed behavior has corresponding test coverage
3. Project builds/imports cleanly

If tests fail: fix before staging. Do not commit broken state.

---

### Phase 3 — Commit

#### 3a. Stage files

```bash
git add <specific-files>    # never use `git add .` blindly
git diff --cached --stat    # confirm staged contents match intent
```

#### 3b. Construct the commit message

Follow [commit-format](./references/commit-format.md) for full format spec. Key rules:
- Subject: `<type>(<scope>): <description>` — max 72 chars
- Allowed types: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore`, `revert`
- Body sections (in order): `## Intent`, `## Reasoning`, `## Scope`, `## Narrative`
- Agent trailers: `Spec-Ref`, `Task-Ref`, `Agent`, `Pipeline`, `Edited`
- Use `None` for `Spec-Ref` and `Task-Ref` when no spec/task applies

#### 3c. Execute the commit

```bash
git commit -m "<subject>" -m "<body-with-sections>" -m "<trailers>"
# — or use a temp file for multi-section messages:
git commit -F /tmp/commit_msg.txt
```

---

### Phase 4 — Pipeline Resolution (dev-stack repos only)

After `git commit`, the dev-stack hooks fire automatically. Two hooks run:

**`pre-commit`** (stages 1–2: lint, typecheck)
- Lint may **auto-format** Python files → `git status` will show modified files
- If reformatted files appear: `git add <reformatted-files>` and `git commit --amend --no-edit`

**`prepare-commit-msg`** (stages 3–9: test → security → docs-api → docs-narrative → infra-sync → visualize → commit-message)
- See [dev-stack-pipeline](./references/dev-stack-pipeline.md) for the **full artifact checklist**

#### 4a. Check pipeline health

```bash
cat .dev-stack/pipeline/last-run.json    # review stage results
# Alternatively:
dev-stack --json status
```

Look for:
- Any `"status": "fail"` on a HARD stage → must fix before proceeding
- Any `"status": "warn"` on SOFT stages → review and address if relevant
- Any stage with `"status": "skip"` where you expected a pass → investigate (missing tool? silent failure?)

#### 4b. Stage pipeline-generated artifacts

After the pipeline runs, stage any auto-generated artifacts that are not gitignored:

```bash
git status    # look for new/modified files from the pipeline
# Common artifacts to stage:
git add .secrets.baseline          # if security stage updated it
git add docs/api/*.rst              # if docs-api ran
git add docs/guides/index.md       # if docs-narrative ran (direct mode)
git add .dev-stack/pipeline/last-run.json   # if tracked
```

If there are new staged files: `git commit --amend --no-edit` to fold them into the commit.  
If the pipeline **rewrote the commit message** (commit-message stage): verify the new message is correct, adjust if needed.

#### 4c. Resolve hard-gate failures

| Stage | Failure | Fix |
|---|---|---|
| `lint` | ruff format or check failed | `ruff format .` then `ruff check --fix .`, re-stage |
| `typecheck` | mypy errors | Fix type errors in source, re-run pipeline |
| `test` | pytest failures | Fix failing tests, re-run pipeline |
| `security` | Unaudited secrets | Audit via `detect-secrets audit .secrets.baseline` then re-run |
| `docs-api` | Sphinx build error | Fix RST/docstring errors, re-run |

---

### Phase 5 — Rebase on origin/main

```bash
git fetch origin
git rebase origin/main
```

If conflicts arise:
1. Resolve conflict markers in each file
2. `git add <resolved-file>`
3. `git rebase --continue`
4. Re-run tests after rebase to confirm nothing broke

---

### Phase 6 — Open PR

```bash
# Using GitHub CLI
gh pr create \
  --title "<conventional-commit-subject>" \
  --body "<intent-section-from-commit-body>" \
  --base main

# The PR body should include:
# - Intent (why this change)
# - Spec-Ref / Task-Ref if applicable
# - Pipeline status summary
```

Note the PR URL/number for the review phase.

---

### Phase 7 — Wait for Copilot PR Review

After opening the PR, wait for the automated Copilot review to complete. Poll with:

```bash
gh pr checks <pr-number> --watch    # wait for CI + Copilot review checks
gh pr view <pr-number> --comments   # read review comments when ready
```

Or check the PR in the browser. Do not proceed until the review is complete.

---

### Phase 8 — Resolve PR Comments

For **each** review comment or suggestion:

1. **Read** the comment carefully — understand the concern
2. **Evaluate** validity:
   - Is the suggestion technically correct?
   - Does it align with the project's constitution (if one exists)?
   - Does it improve correctness, readability, or safety?
3. **Implement** valid suggestions — follow the same TDD rules (add/update tests if behavior changes)
4. **Reject** invalid suggestions — add a reply explaining why (do not silently ignore)

After implementing all valid changes:

```bash
git add <changed-files>
git commit    # new commit following Conventional Commits format
# type: fix or refactor, scope: same as original, description: "address PR review"
# Body sections + trailers as normal
```

Push the fix commit(s):

```bash
git push origin HEAD
```

Repeat Phase 7–8 until no outstanding unresolved comments.

---

### Phase 9 — Constitution Validation

If a constitution file exists, perform a final review before considering the work done.

```bash
# Locate constitution
cat .specify/memory/constitution.md 2>/dev/null \
  || cat .specify/constitution.md 2>/dev/null \
  || cat specs/constitution.md 2>/dev/null
```

Check the committed changes against **every principle**:

- Architecture rules (layer boundaries, dependency direction, etc.)
- Naming conventions
- Testing requirements (coverage thresholds, TDD mandate)
- Security rules (no hardcoded secrets, input validation, etc.)
- Documentation standards
- Pipeline / workflow rules

**Any violation** must be corrected in a new fix commit before the PR is merged. Constitution compliance is non-negotiable.

---

### Phase 10 — Sync Local Main

After the PR is merged, bring local `main` in sync with `origin/main`:

```bash
git checkout main
git fetch origin
git rebase origin/main
```

If git reports divergent branches (local commits that are now duplicated by the merge commit), `rebase` will detect and skip the already-applied commits — this is expected and safe. Confirm with:

```bash
git log --oneline -5   # HEAD should match origin/main
git status             # should read "up to date"
```

If the feature branch is no longer needed:

```bash
git branch -d <feature-branch>
```

## Quick Checklist

```
[ ] git status / diff reviewed — know exactly what changed
[ ] .gitignore updated for any untracked junk files
[ ] Changes form a single logical unit (atomic)
[ ] Tests pass (pytest or equivalent)
[ ] New behavior has test coverage
[ ] Correct files staged (source + tests + config, no build artifacts/secrets)
[ ] Commit message: conventional subject ≤72 chars
[ ] Commit message: Intent, Reasoning, Scope, Narrative sections present
[ ] Commit message: Spec-Ref / Task-Ref trailers set (None if not applicable)
[ ] Pipeline ran: all HARD stages pass (dev-stack repos)
[ ] Pipeline artifacts staged and folded into commit (dev-stack repos)
[ ] Rebased on origin/main cleanly
[ ] PR opened
[ ] Copilot review complete
[ ] All valid review suggestions implemented and committed
[ ] Constitution validated (if exists)
[ ] Local main rebased on origin/main and up to date
[ ] Feature branch deleted (if no longer needed)
```
